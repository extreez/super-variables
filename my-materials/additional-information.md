Отлично, распишу как это могло бы выглядеть на практике.

## 1) Smart Import Merge (импорт с diff)

Проблема: обычный импорт ломает существующие токены.

Как сделать:

- При загрузке JSON плагин строит сравнение с текущими Variables.
- Категории: new, changed, deleted, renamed, conflicts.
- Перед применением показывается превью.

Пример:

- В Figma есть color/brand/primary = #4F46E5
- В файле импорта: color/brand/primary = #4338CA
- UI: “Changed value”
- Кнопки: Accept, Keep local, Accept all changed

Для rename:

- Было space/md, стало spacing/md
- Плагин предлагает “распознан возможный rename”, а не удалить+создать.

---

## 2) Alias Graph + валидация

Проблема: трудно понять, откуда берётся итоговое значение.

Как сделать:

- Граф зависимостей токенов (узлы и связи).
- Проверки:
    - циклы (A -> B -> C -> A)
    - битые ссылки (color/text/default -> color/neutral/900, которого нет)
    - “висячие” токены (никто не использует)

Пример:

- button/bg/default -> color/brand/500
- color/brand/500 -> palette/indigo/600
- Dev в панели видит цепочку и финал: #4F46E5

---

## 3) Мультиформатный экспорт

Проблема: дизайн и код живут в разных форматах.

Как сделать:

- Один источник (Figma Variables), несколько target-форматов:
    - DTCG JSON
    - CSS custom properties
    - Tailwind theme.extend
    - SCSS maps
    - Android XML / iOS Swift enum (MVP можно позже)

Пример (из одного токена color/brand/500):

- CSS: --color-brand-500: #4F46E5;
- Tailwind: "brand": { "500": "#4F46E5" }
- DTCG:

`{ "color": { "brand": { "500": { "$type": "color", "$value": "#4F46E5" } } } }`

---

## 4) Round-trip sync (двусторонняя синхронизация)

Проблема: после экспорта правки в коде теряются для дизайна.

Как сделать:

- Каждому токену присвоить стабильный tokenId (метаданные).
- При повторном импорте матчить не только по имени, но и по tokenId.
- Rename не ломает связь.

Пример:

- Был space/md с tokenId=tok_142
- В репо переименовали в spacing/md
- Импорт находит tok_142 и обновляет имя корректно.

---

## 5) Token Linting (правила качества)

Проблема: хаотичный нейминг и неполные mode.

Как сделать:

- Набор правил:
    - kebab-case или slash-case
    - запрет final, new2, temp
    - обязательные mode: light и dark
    - запрет raw hex в semantic tokens
- Автофикс там, где возможно.

Пример отчёта:

- color/button/primary: missing dark mode
- spacing/Large: invalid naming (предложение: spacing/large)

---

## 6) Theme Matrix (редактор mode’ов в таблице)

Проблема: mode редактируются неудобно и по одному.

Как сделать:

- Таблица: строки = токены, колонки = mode (light, dark, hc).
- Массовые операции:
    - “дублировать light -> dark”
    - “применить multiply +10% к всем dark color tokens”

Пример:

- Выделил группу color/surface/*
- Действие “darken 8% для dark mode”
- Получил пакетное обновление сразу десятков значений.

---

## 7) Semantic Layer (primitive vs semantic)

Проблема: напрямую используют палитру, а не смысловые токены.

Как сделать:

- Два слоя:
    - primitive: palette/blue/500
    - semantic: color/text/primary, color/bg/canvas
- Плагин показывает нарушения: компонент ссылается на primitive напрямую.

Пример:

- Найдено: button/bg = palette/blue/500
- Рекомендация: button/bg -> color/action/primary

---

## 8) Dev handoff panel

Проблема: разработчику нужен “готовый ответ”.

Как сделать:

- Для выбранного токена показывать:
    - итоговое значение
    - цепочку алиасов
    - snippet для платформы

Пример:

- Токен: radius/card/lg
- Вывод:
    - Final: 12px
    - CSS: border-radius: var(--radius-card-lg);
    - iOS: .radiusCardLg

---

## 9) Safe Migration (безболезненные миграции)

Проблема: большой рефакторинг ломает проекты.

Как сделать:

- Режим “shadow migration”:
    - создаются новые токены
    - старые становятся alias на новые
    - после периода стабилизации — cleanup

Пример:

- Старый: color/primary
- Новый: color/action/primary
- Временно: color/primary -> color/action/primary

---

## 10) Usage Analytics

Проблема: неясно, что реально используется.

Как сделать:

- Метрики:
    - usage count по токенам
    - “legacy tokens”
    - покрытие semantic слоем
- Фильтр по странице/файлу/библиотеке.

Пример:

- color/legacy/blue используется в 2 слоях
- Статус: “кандидат на замену”
- Кнопка: “Find instances”
